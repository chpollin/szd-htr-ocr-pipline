# SZD OCR/HTR Pipeline

> **Experimental sub-project of [Stefan Zweig Digital](https://stefanzweig.digital/).**
> This project was built entirely using the **[Promptotyping](https://lisa.gerda-henkel-stiftung.de/digitale_geschichte_pollin)** method: all code, the pipeline architecture, the frontend, and the documentation were generated by [Claude Code](https://docs.anthropic.com/en/docs/claude-code) (Anthropic, Opus 4.6). The project lead ([Christopher Pollin](https://github.com/chpollin)) acted as **project manager and domain expert** — defining requirements, evaluating results, making directional decisions — rather than as a software developer in the traditional sense. This project should be understood as a methodological experiment: it explores how far a single researcher can go in building a complete Digital Humanities pipeline through AI-assisted promptotyping.

VLM-based handwritten text recognition (HTR) and OCR for the Stefan Zweig estate at the [Literaturarchiv Salzburg](https://www.literaturarchiv.at/). The pipeline produces machine-readable text from 2,107 digitised facsimiles (manuscripts, correspondence, typescripts, personal documents) using Google's Gemini 3.1 Flash Lite.

**[Live Viewer & Catalogue](https://chpollin.github.io/szd-htr-ocr-pipeline/)** — browse all transcriptions with facsimile comparison, quality signals, and search.

## Approach

The pipeline combines a 4-layer prompt system with automated quality assessment:

1. **Context resolution** — TEI-XML metadata provides title, language, and object type. One of 9 prompt groups is assigned automatically (handwriting, typescript, form, short text, tabular, galley proof, convolute, newspaper clipping, correspondence).
2. **VLM transcription** — All facsimile images plus a tailored prompt are sent to Gemini 3.1 Flash Lite. Large objects (>20 images) are automatically chunked and merged.
3. **Quality signals** — 7 automatic signals (page-type classification, marker density, duplicate detection, language consistency, etc.) flag objects for manual review.
4. **Verification** — Cross-model consensus (Flash Lite + Flash + Claude as judge) and agent-based checks comparing transcription against the facsimile images.

Diplomatic transcription without normalisation. Markup: `[?]` uncertain, `[...]` illegible, `~~...~~` struck through, `{...}` insertion.

## Data

2,107 digitised objects with 18,719 facsimile scans (~23 GB), fully available in a local backup with metadata.

| Collection | Objects | Images | Images/Obj (median) | TEI source |
|---|---|---|---|---|
| Personal documents | 127 | 2,879 | 3 | [TEI](https://stefanzweig.digital/o:szd.lebensdokumente/TEI_SOURCE) |
| Works (manuscripts) | 169 | 7,842 | 21 | [TEI](https://stefanzweig.digital/o:szd.werke/TEI_SOURCE) |

| Correspondence | 1,186 | 4,154 | 3 | Backup metadata |

Languages: German (95.6%), English (2.6%), French (1.5%), Italian, Spanish (based on TEI metadata analysis).

Image format: JPEG, approx. 4,800 × 7,200 px.

## Setup

Python 3.10+ (tested with 3.11). Requires a [Google AI API key](https://ai.google.dev/).

```bash
pip install -r requirements.txt
# Create .env with: GOOGLE_API_KEY=AIza...
```

Facsimile images are served via [GAMS](https://gams.uni-graz.at/) (University of Graz) and do not need to be stored locally.

## Usage

```bash
# Transcribe a single object
python pipeline/transcribe.py o_szd.161 -c lebensdokumente

# Transcribe an entire collection
python pipeline/transcribe.py -c lebensdokumente

# Transcribe everything
python pipeline/transcribe.py --all

# Dry run (list without API calls)
python pipeline/transcribe.py --all --dry-run

# Rebuild viewer data
python pipeline/build_viewer_data.py

# Local dev server with review API
python pipeline/serve.py
```

Results are stored as JSON in `results/{collection}/`. Two export formats: **Page-JSON v0.2** (internal working format with descriptive metadata) and **METS/MODS + PAGE XML** (archival exchange format, compatible with GAMS, Transkribus, eScriptorium, OCR-D). Pipeline script documentation: [`pipeline/README.md`](pipeline/README.md). Result file documentation: [`results/README.md`](results/README.md).
